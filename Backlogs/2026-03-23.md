## Key Architecture Decisions

1. **Full GhosttyKit** over VT-only — eliminates 1,700+ LOC of custom rendering
2. **Write callback** for terminal→SSH — intercepts at `Surface.queueIo()` before NullPty, zero impact on non-iOS builds
3. **GhosttyKitSurfaceIO** adapter — thread-safe (NSLock), callback fires from IO thread
4. **ShellSessionProvider** protocol — transport-agnostic, enables future local shell or other backends
5. **Config via temp file** — `ghostty_config_load_file()` for full theme/font/cursor mapping

## Data Flow

```
SSH → Display:
  SSH server → swift-nio-ssh → InteractiveShell.output
    → SSHPTYBridge → TerminalIO.write()
    → GhosttyKitSurfaceIO → ghostty_surface_process_output()
    → VT parser → Metal renderer → CAMetalLayer

Display → SSH:
  User types on keyboard → UIKeyInput / pressesBegan
    → ghostty_surface_key() / ghostty_surface_text()
    → Surface.queueIo() → write callback (IO thread)
    → GhosttyKitSurfaceIO._outputHandler (NSLock-guarded)
    → SSHPTYBridge.sendToShell()
    → InteractiveShell.write() → SSH server
```

## Known Risks & Caveats

- **NullPty on iOS**: Ghostty uses NullPty (fd=0, all ops no-op). Our write callback intercepts BEFORE NullPty, so it works, but other Termio features may have edge cases.
- **Config API**: Using temp-file-based config loading (`ghostty_config_load_file`). If GhosttyKit adds programmatic `ghostty_config_set()`, we should switch.
- **Threading**: Write callback fires from GhosttyKit's IO thread — `GhosttyKitSurfaceIO` uses NSLock, not actors, because the callback is a C function pointer.
- **Termio initialization**: With NullPty, the Exec backend's behavior on iOS is untested. May need investigation if rendering has issues.
